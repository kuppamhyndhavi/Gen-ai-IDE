# 📖 MySQL Database Migration Guide for AI Code Editor

This document outlines how to switch the application's backend from MongoDB to **MySQL** using **SQLAlchemy** (an Object-Relational Mapper) in Python. 

## 1. What is MySQL & Why Use It?
**MySQL** is a Relational Database Management System (RDBMS). Unlike MongoDB (which stores flexible, unpredictable JSON-like "documents"), MySQL stores data in strict, highly structured **Tables** consisting of **Rows** (records) and **Columns** (attributes).

**Why use it in this project?**
While MongoDB is great for quickly dumping unstructured AI chat logs, a relational database like MySQL is superior when your application scales to include **Relationships**. 
If you want to add **User Accounts**, **Billing Systems**, or **Saved Projects (Workspaces)** to your IDE, MySQL enforces strict rules (Foreign Keys) ensuring that a "Chat Session" cannot exist without a valid "User Account" tied to it. This guarantees data integrity.

---

## 2. The Database Schema (Architecture for this App)
In MySQL, we design a **One-to-Many Relationship**. One `Session` can have Many `Messages`.

**Table 1: `sessions`**
| Column Name | Data Type | Constraint | Description |
| :--- | :--- | :--- | :--- |
| `id` | VARCHAR(255) | **Primary Key** | The unique `session-123456` string generated by React. |
| `created_at` | DATETIME | NOT NULL | When the session started. |

**Table 2: `chat_messages`**
| Column Name | Data Type | Constraint | Description |
| :--- | :--- | :--- | :--- |
| `id` | INTEGER | **Primary Key** | Auto-incrementing ID. |
| `session_id` | VARCHAR(255) | **Foreign Key** | References `sessions.id`. Links the message to the chat. |
| `role` | VARCHAR(50) | NOT NULL | Either 'user' or 'assistant'. |
| `content` | TEXT | NOT NULL | The actual text/code of the message. |
| `timestamp` | DATETIME | NOT NULL | When the message was sent. |

## 3. How It Is Implemented (The Code)

In a FastAPI application, we use **SQLAlchemy** to translate our Python classes directly into MySQL tables. We will also use `aiomysql` to ensure the database calls are asynchronous, perfectly matching FastAPI's performance model.

### Step 3A: Install Dependencies
Remove `motor` (MongoDB) and install SQLAlchemy along with the asynchronous MySQL driver (`aiomysql`) and a cryptography package often required by MySQL connectors.
```bash
pip install sqlalchemy aiomysql cryptography
```

### Step 3B: Update the Environment Variables (`.env`)
You must configure the connection engine to point to your local or remote MySQL server.
```ini
# Remove MONGO_URL
# Add MYSQL_URL
DATABASE_URL="mysql+aiomysql://root:yourpassword@localhost:3306/ide_database"
```

### Step 3C: Database Setup (`database.py`)
Configure the asynchronous engine that talks to the MySQL database.
```python
import os
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession
from sqlalchemy.orm import declarative_base, sessionmaker

DATABASE_URL = os.getenv("DATABASE_URL", "mysql+aiomysql://root:password@localhost:3306/ide_database")

# Create the Async Engine
engine = create_async_engine(DATABASE_URL, echo=True)

# Create a Session Factory (how we perform transactions)
async_session = sessionmaker(
    engine, class_=AsyncSession, expire_on_commit=False
)

# Base class for our Models
Base = declarative_base()
```

### Step 3D: Define the Models (Schemas)
This replaces the concept of MongoDB Collections. We define our MySQL tables as Python classes.
```python
from sqlalchemy import Column, Integer, String, DateTime, Text
import datetime

class ChatMessage(Base):
    __tablename__ = "chat_messages" # The name of the MySQL table

    id = Column(Integer, primary_key=True, index=True, autoincrement=True)
    session_id = Column(String(255), index=True) # Max length is required for MySQL VARCHAR
    role = Column(String(50)) 
    content = Column(Text) # Use Text for large AI code outputs
    timestamp = Column(DateTime, default=datetime.datetime.utcnow)
```

### Step 3E: Rewrite CRUD Operations in `server.py`
We must update the FastAPI endpoints to use SQLAlchemy logic instead of MongoDB logic.

#### 1. Creating/Saving a Message
*How it works: We instantiate the Python class and add it to the DB session, then commit it.*
```python
async def save_message_to_db(session_id: str, role: str, content: str):
    async with async_session() as session:
        new_msg = ChatMessage(
            session_id=session_id, 
            role=role, 
            content=content
        )
        session.add(new_msg)
        await session.commit() 
```

#### 2. Reading/Fetching Chat History
*How it works: We use a `select` statement to query rows where the `session_id` matches, ordering by time.*
```python
from sqlalchemy.future import select

async def get_recent_chat_history(session_id: str, limit: int = 15):
    async with async_session() as session:
        # Build the Query: SELECT * FROM chat_messages WHERE session_id = ? ORDER BY timestamp
        stmt = select(ChatMessage).where(ChatMessage.session_id == session_id).order_by(ChatMessage.timestamp)
        
        result = await session.execute(stmt)
        messages = result.scalars().all()
        
        # Format the result back into the standard list of dictionaries the AI expects
        return [{"role": msg.role, "content": msg.content} for msg in messages[-limit:]]
```

#### 3. Deleting Chat History
*How it works: We use a `delete` statement to remove all rows associated with a specific session.*
```python
from sqlalchemy import delete

@api_router.delete("/chat/clear/{session_id}")
async def clear_chat(session_id: str):
    async with async_session() as session:
        # Build Query: DELETE FROM chat_messages WHERE session_id = ?
        stmt = delete(ChatMessage).where(ChatMessage.session_id == session_id)
        
        await session.execute(stmt)
        await session.commit() # Commit the deletion transaction
        return {"status": "cleared"}
```

## 4. Run Migrations (Creating the Tables)
Unlike MongoDB, MySQL will throw an error if you try to insert data into a table that doesn't exist yet. Before running the app, you must tell SQLAlchemy to physically create the tables in MySQL based on your Python classes.

You can do this by executing a script once at startup:
```python
async def init_db():
    async with engine.begin() as conn:
        # This executes the CREATE TABLE statements in MySQL
        await conn.run_sync(Base.metadata.create_all)
```

## 5. Summary: MySQL vs MongoDB for this Project

| Feature | MongoDB (Current) | MySQL / SQLAlchemy (Proposed) |
| :--- | :--- | :--- |
| **Data Structure** | Highly flexible (JSON). AI output can be wildly unpredictable. | Highly rigid. You must guarantee the AI output fits the exact column types (e.g., using `TEXT` for code). |
| **Setup Speed** | Very fast. Just dump data in. No migrations needed. | Slower. Requires defining Models and running `create_all()` or Alembic migrations if the structure changes. |
| **Relationships** | Requires manual, complex `$lookup` queries to link Users to Sessions. | Excellent. Built for this. Foreign Keys ensure a Chat Session cannot exist without a User account. |
| **Best For...** | Rapid prototyping, storing unpredictable large text blobs. | Production systems requiring Authentication, billing, and strict data integrity. |
